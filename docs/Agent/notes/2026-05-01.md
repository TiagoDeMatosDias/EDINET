## Round 2 — Drag-and-drop column reorder + session cache

### Feature 3: Drag-and-drop column reordering (no re-run)

Column bars (both regular columns and computed columns) now have a drag grip (⠿) and support HTML5 drag-and-drop to reorder. Reordering does NOT trigger a new API call — it only rearranges the existing results table via `reorderResults()`.

**Implementation**:
- `_makeDraggable(bar, kind, idx, onReorder)` — shared drag-and-drop helper that handles `dragstart`, `dragend`, `dragover`, `dragleave`, `drop` events. Tracks kind ('col' or 'comp') to prevent cross-section drops.
- `renderColumns()` — now uses `.forEach((col, i) => ...)` to track index, adds drag grip span, wires `_makeDraggable`
- `renderComputedColumns()` — same treatment
- `reorderResults()` — builds desired column order from `ST.columns` refs + `ST.computedColumns` names, maps old column positions to new ones (with case-insensitive fallback), reorders both `ST.results.columns` and `ST.results.rows`, then re-renders the table. Any columns not in the desired list (e.g. LatestPrice, PriceDate) are appended at the end.
- Removing a column or computed column also calls `reorderResults()` to update the table

**CSS additions**:
- `.scr-col-grip` — muted drag handle, cursor:grab
- `.scr-col.is-dragging` — opacity 0.4 when being dragged
- `.scr-col.is-over` — accent border + background when hovering a drop target

### Feature 4: Session cache (survives page navigation)

When the user navigates away from Screening and comes back, the previous run results and all configuration are restored — no re-run needed.

**Implementation**:
- `cacheState()` — serialises key ST fields (dbPath, availableMetrics, screeningDate, criteria, columns, computedColumns, results, sqlDisplay, formattedValues, etc.) to `sessionStorage` under `'screening_state'`
- `restoreCachedState()` — reads from sessionStorage, rebuilds ST state. Returns `true` if valid cache found.
- `init()` — tries `restoreCachedState()` first. If cached results exist, renders them immediately and refreshes metrics in background. Otherwise loads from DB as before.
- `run()` — calls `cacheState()` after successful API response
- `wireShell()` — registers `beforeunload` listener for belt-and-suspenders caching
- Uses `sessionStorage` (not localStorage) so cache is scoped to the browser tab and doesn't persist across sessions
