## Frontend Bug Fixes — Orchestrator View

### Bug 1: Setup Load instantly resets pipeline
**Root cause**: `showLoadMenu` in `orchestrator/index.js` called `callbacks.setView?.('orchestrator')` after `hydrateSetup()`. This triggered `window.location.href = '/orchestrator'` — a full page reload even when already on that page. The reload ran `initializeSetup()` which reset `STATE.pipeline = []`.

**Fix**: Removed `callbacks.setView?.('orchestrator')` from the load menu item click handler. The user is already on the orchestrator page; no navigation needed.

### Bug 2: Console footer "Hide" button not working
**Root cause**: Double event listener registration on `#console-toggle`. Both `wireTopbarEvents()` (topbar.js) and orchestrator.js's `bootstrap()` added click handlers that both toggled `console-collapsed`. Two synchronous handlers → first toggles, second toggles back → net no change.

**Fix**: Removed manual console button wirings from orchestrator.js. `wireTopbarEvents()` is the single source of truth for both console footer toggle and (on other pages) top bar toggle. Also removed unused `els.toggleConsoleBtn` and `els.consoleToggle` from DOM cache.

### Bug 3: Top bar console button shouldn't exist on orchestrator
**Fix**: Removed `<button id="toggle-console-btn">` from orchestrator.html top bar. Console toggle is now only via the footer button. Other pages (main, screening, security) still have their top bar buttons.

## Round 3 — File upload support for Import Stock Prices

### Problem
When the user picks a CSV file via the system file picker in the "Import Stock Prices" step, only the **filename** was captured. The backend received just a filename string and tried to `pd.read_csv(filename)` which failed because there's no file at that path on the server.

### Solution
**Frontend** (`orchestrator/index.js`):
- `createFilesystemInput` now detects `field_type === 'file'` and reads file content as base64 via `FileReader.readAsDataURL()`
- Stores `{filename, content}` (base64 payload) in the config instead of a plain string
- Text input still shows just the filename for display
- Added `readFileAsBase64()` helper function
- `pickPathLikeValue()` extended with `wantsFileContent` flag — for file-type fields, reads the actual file content from both `showOpenFilePicker` (File System Access API) and fallback `<input type="file">`
- CSV file type filter added to the file picker dialog
- Non-file fields (database, directory, path-like keys) continue to store plain string paths

**Backend** (`orchestrator/orchestrator.py`):
- Added `resolve_file_uploads()` — a central resolver that processes the raw config dict before validation
- For every enabled step, looks up its `StepDefinition.input_fields`, finds fields with `field_type == "file"`, and if the value is a dict with `content`, decodes base64, writes to a temp file, and replaces the value with the temp file path
- Called from both `validate_input()` and `run()` so it works regardless of entry point
- All 4 steps with `field_type == "file"` (import_stock_prices_csv, backtest_set, parse_taxonomy, populate_company_info) benefit automatically

**import_stock_prices_csv.py**:
- Added belt-and-suspenders fallback: also checks for dict format at the step level and decodes content to temp file if the central resolver didn't run

### Files modified
- `src/web_app/frontend/orchestrator/index.js` — createFilesystemInput + pickPathLikeValue + readFileAsBase64
- `src/orchestrator/orchestrator.py` — resolve_file_uploads function
- `src/orchestrator/import_stock_prices_csv/import_stock_prices_csv.py` — fallback file upload handling
- `tests/test_orchestrator.py` — 6 new tests for resolve_file_uploads

## Round 4 — New screening criteria: vs Stock Price

### Feature
Added a new `stock_price` comparison mode to the screening criteria builder. This allows comparing any financial metric column against the most recent stock price, with an optional arithmetic expression on the left side.

Example: `[PerShare].[Sales] / 2 <= [Current Price]`

### Implementation

**Backend** (`src/screening/screening.py`):
- Added `_validate_stock_price_expr()` — whitelist validator for left-side arithmetic expressions (only digits, + - * / . ( ) spaces allowed)
- Added `comparison_mode == "stock_price"` branch in `build_screening_query` WHERE clause builder
- SQL generated: `(alias.[col] expr) OP s_p.[Price]` where `s_p.[Price]` is already the latest stock price (resolved via `sp_max`/`s_p` join chain)
- Rejects BETWEEN operator

**Frontend** (`src/web_app/frontend/screening/index.js`):
- Added "vs Stock Price" option to criteria builder mode selector
- Expression bar renders `[Current Price]` as a stylised italic token on the right side
- Left-side expression is an editable inline token (click to edit)
- Operator menu and builder operator selector restrict to `> >= < <= = !=` when in stock_price mode
- `run()` function passes `left_expression` to the API

**CSS** (`src/web_app/frontend/common/styles.css`):
- Added `.scr-tok-price` style for the `[Current Price]` token (accent colour, italic)

## Round 3 — Fix `get_tickers_from_prices()` sqlite_master check bug

### Bug
`get_tickers_from_prices()` in `src/orchestrator/update_stock_prices/update_stock_prices.py` checked table existence via:
```python
cursor.execute("SELECT name FROM sqlite_master WHERE type='table' AND name=?", ("CompanyInfo",))
if not cursor.fetchone():
    return []
```
The `sqlite_master.name` comparison is case-sensitive — if the table was created with different casing (e.g. by pandas `to_sql`), the check fails and the function returns `[]` even when the table exists and has data.

### Fix
- Replaced the fragile `sqlite_master` pre-check with `try/except sqlite3.OperationalError` around the actual query — catches both missing table and missing column
- Added `table_name` parameter (default `"CompanyInfo"`) so the function works with both `CompanyInfo` (column `Company_Ticker`) and `Stock_Prices` (column `Ticker`)
- Fixed docstring (wrongly referenced `prices_table`)
- Updated `update_all_stock_prices()` signature to accept `Company_Table`, `prices_table`, `standardized_table` parameters with defaults
- Updated `run_update_stock_prices()` to read `DB_COMPANY_INFO_TABLE`, `DB_STOCK_PRICES_TABLE`, `DB_FINANCIAL_DATA_TABLE` from config and pass them through
- Changed flow: try `prices_table` first (for existing tickers with price data), fall back to `Company_Table` if empty

### Tests added (8 new tests in `tests/test_orchestrator.py::TestGetTickersFromPrices`)
- `test_returns_empty_when_table_does_not_exist` — in-memory DB, no table
- `test_returns_empty_when_table_exists_but_has_no_ticker_column` — table exists without the expected column
- `test_returns_empty_when_all_tickers_are_null_or_whitespace` — all values are NULL, empty, or spaces
- `test_returns_distinct_non_empty_tickers_from_company_info` — deduplicates + filters NULL/empty
- `test_returns_tickers_from_stock_prices_table` — works with `Stock_Prices` (column `Ticker`)
- `test_trims_whitespace_from_tickers` — TRIM handles leading/trailing spaces but preserves them for DISTINCT
- `test_table_exists_but_with_different_casing_in_sqlite_master` — the core bug: table exists as `companyinfo` (lowercase) but query uses `CompanyInfo`; now handled via OperationalError catch
- `test_default_table_name_is_company_info` — backward compat: omitting table_name defaults to CompanyInfo

### Files modified

## Round 4 — Database split refactoring (hardcoded DB paths per step)

### Background
Every orchestrator step was individually selecting its database via step config fields (`Target_Database`, `Source_Database`, `Database`). This was error-prone and confusing — users had to remember which DB to point each step at. Some steps needed 2 databases.

### Design
Split into 2 databases with a config file (`config/database_paths.json`):
- **DB1** (`data/databases/Base.db`): `DocumentList`, `financialData_full` — raw ingested data
- **DB2** (`data/databases/Standardized.db`): Everything else — `Taxonomy`, `CompanyInfo`, `FinancialStatements`, `IncomeStatement`, `BalanceSheet`, `CashflowStatement`, `ShareMetrics`, ratio tables, rolling tables, `Stock_Prices`
