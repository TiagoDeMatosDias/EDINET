## Screening View — "Update Prices" button

Added a button to the Screening view that updates stock prices for all companies
in the current screening results.

### Implementation

**Backend** (`src/web_app/api/screening.py`):
- Added `POST /api/screening/update-prices` endpoint
- Accepts `db_path` and `tickers` array
- Calls `update_security_price()` from `src.security_analysis` for each ticker
- Returns per-ticker results with `ok`, `rows_inserted`, and `message`

**Frontend** (`src/web_app/frontend/screening/index.js`):
- Added "Update Prices" button in the results toolbar (left of "Export CSV")
- Extracts tickers from the results table by finding a Ticker/Company_Ticker column
- Shows progress ("Updating…"), disables button during request
- On success, auto re-runs the screening so price-dependent computed columns (P/E, etc.) reflect updated prices
- If no results or no ticker column, shows appropriate warnings

**Tests** (`tests/test_screening_api.py`):

- All 25 screening API tests pass

## Security fixes — drill-down link + URL leak

### Issue: Screening drill-down leaked filesystem path in URL
The `drill()` function in screening/index.js appended `db_path` (filesystem path like
`C:\...\database.db`) to the security-analysis page URL query string. This exposed
the server's filesystem structure in browser history, server logs, and network traces.

### Issue: Drill-down link was broken on security_analysis page
The `security.js` entry point only restored company from `sessionStorage`, never from
URL params. When a user clicked a row in screening results, the URL contained
`edinet_code` but the security page ignored it — so no company loaded.

### Fixes applied
- **screening/index.js** (`drill()`): Removed `db_path` from URL params entirely.
  Now writes `edinet_code` to `sessionStorage('sa.lastEdinetCode')` so the security
  page can still pick it up even if URL params fail.
- **security_analysis/security.js**: Now reads `edinet_code` from URL params first
  (`new URLSearchParams(window.location.search)`), falling back to sessionStorage.

### Security audit of frontend storage
- **sessionStorage**: screening cache stores `dbPath`, `availableMetrics`, criteria,
  results. The `dbPath` is already sent to the server in every API call, so this is
  acceptable (same-origin, tab-scoped). No other sensitive data found.
- **localStorage**: orchestrator setups may contain user-configured paths — by design.
- **URL params**: only `edinet_code` and `ticker` now — no filesystem paths.
- All 418 tests pass.
